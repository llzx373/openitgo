# 大章节分段加载评估（TODO #55）

日期：2026-07-17
结论：**不做**

## 样本

| 样本 | 章节 HTML 大小 | 段落数 | 来源 |
|---|---|---|---|
| 合成大章节 | 465 KB（476,142 B；epub 477,245 B） | 4000 | `scripts/make-big-chapter-epub.py`（/tmp/big-chapter.epub） |
| 对照（合成小章） | 116 KB（119,142 B） | 1000 | 同脚本（/tmp/big-chapter-1000.epub） |
| 对照（合成超大章） | 930 KB（952,142 B） | 8000 | 同脚本（/tmp/big-chapter-8000.epub） |

spec 样本阈值为"≥500KB HTML 或 ≥3000 段落"：4000 段样本满足"≥3000 段落"腿，
8000 段对照同时满足"≥500KB HTML"腿。注：计划中"4000 段落 ≈ 520KB"的估算
偏大，实测 119 B/段 → 465KB，故补 8000 段对照覆盖体积腿。

## 测量方法

- release 构建（`cargo build --release`），诊断 harness 启动即打开样本书
  （1280×800 窗口、默认设置：单页分页 / 16px / system-ui）。
- 首次布局耗时：webview 创建 → 首个 position IPC（分页完成、总页数上报）；
  正文绘制与 IPC 同帧量级，另以全屏截图确认正文可读。
- resize 重排耗时：窗口 1280→900 改宽，JS watcher（20ms 轮询）测真实 resize
  处理器完成时刻（列宽样式变更 + `lastLayoutParams` 更新），含 200ms 防抖。
- 内存：`ps` RSS 每 0.2s 采样，app 进程与全部 com.apple.WebKit.* 进程分别
  合计，记录打开前 / 布局后稳态 / resize 后。
- 机器：Mac mini，Apple M4，16 GB，macOS 26.5.2（25F84）。每样本 2~3 轮。
- 会话限制：测量在屏幕休眠、窗口隐藏（visibilityState=hidden）的会话中进行；
  隐藏页的后台定时器钳制（setTimeout 最小 ~1s）会把 200ms 重排防抖放大到
  ~1s，产生的 doneMs 离群值是会话假象而非真实体验；亮屏可见窗口复测予以校正。

## 结果

| 指标 | 合成大章节（4000 段 / 465KB） | 对照（1000 段 / 116KB） | 对照（8000 段 / 930KB） |
|---|---|---|---|
| 首次布局耗时（webview→分页完成） | 233 / 236 / 233 ms（亮屏冷显示一轮 535 ms） | 174 / 171 ms | 328 / 327 ms |
| 首次布局全程（进程启动→分页完成） | 400~434 ms（亮屏一轮 1052 ms） | 346 / 364 ms | 501 / 495 ms |
| resize 重排耗时（含 200ms 防抖） | 可见窗口 211 / 219 ms；隐藏窗 219~1111 ms（>1s 为钳制假象） | 可见窗口约 220 ms；隐藏窗 577 / 901 ms | 437 ms；隐藏窗 1061 ms |
| 内存（打开后增量，app + WebKit 合计） | ≈ +245 MB（app +129 MB、WebKit +116 MB） | ≈ +219 MB（+124 / +95 MB） | ≈ +272 MB（+134 / +138 MB） |
| resize 后内存变化 | ≈ 0（+1~2 MB，噪声内） | — | — |

## 结论依据

判定阈值（spec）：首次布局 >1s 或内存明显异常 → 建议做分段加载。

实测距阈值裕量充足：

- 最大样本（930KB / 8000 段，已超 spec 双阈值）首次布局核心 328ms、含进程
  启动全程 501ms，均为 1s 阈值的一半以下；4000 段样本 233ms。亮屏冷显示
  一轮 1052ms 也仅擦阈值边缘，且含 caffeinate 唤醒显示后的 GPU 冷路径。
- 耗时构成：固定开销（阅读 shell 加载、章节 fetch、`render_chapter_html`、
  JS DOM 构建）约 150ms；随章节规模的边际成本约 22ms/1000 段
  （171 → 233 → 328ms，近似线性）。CSS columns 全量布局本身由 WebKit 同步
  完成，4000 个段落级在数十毫秒量级。
- 重排确为整章重算：`layout()` 重设列样式后读 `scrollWidth`，强制整章同步
  布局，无增量路径；resize 与改字体/字号（applySettings）同路径。实测整章
  重排完成（含 200ms 防抖）211~545ms，落在验收标准 500ms 线附近及以内。
- 内存：基本盘约 215~220MB（webview 基础设施、GPU、字体缓存），随章节大小
  边际约 7.5MB/1000 段，近似线性且斜率平缓；resize 后无增长。无异常。
  首次冷启动（GPU/wgpu 初始化）app RSS 瞬时峰值 438MB，随后回落至 ~133MB，
  与章节大小无关。

两项触发条件（>1s、内存异常）均未命中，故结论为**不做**。

## 若结论为"不做"：留下的观察项

1. 合成样本为纯文本短段落；真实书籍含图片、表格、复杂 CSS，单章布局成本
   更高——#54 真机走查时留意大开本/合集书的首屏时间，若实测 >1s 重启评估。
2. 持续观察真实藏书的最大单章规模（合集型 EPUB 单章可达数 MB）；单章 HTML
   超过 ~2MB 或真机首布局实测 >1s 时重启本评估。
3. 隐藏/后台窗口中重排防抖被系统钳制到 ~1s（本次离群值来源）；真实阅读时
   窗口可见、无影响，若未来支持后台标签页阅读需重新评估该路径。
4. 单页模式下总页数对窗宽近似不变（不换行内容仅列宽缩放），本次测量环境
   无法以页数变化作重排信号——后续若写自动化重排回归，须以列宽样式或
   scrollWidth 变化为信号。
